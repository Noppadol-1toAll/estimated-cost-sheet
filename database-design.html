Database Design — Estimated Cost Sheet

Phase 1: Master Data บน Supabase (PostgreSQL) · อ้างอิงแนวทาง Product System Blueprint (M02 Item Master / M04 Vendor)
ที่มาข้อมูล: (Rev.2.4.7) Feasibility 1toAll (Draft).xlsx 18 ตาราง (master 11 + transaction 7) · 6,328 แถว seed + 14 ผู้ใช้ ปรับปรุง: 12/06/2026

เป้าหมาย Phase 1

ย้าย master data ทั้งหมดที่เคย hardcode ใน index.html (item catalog 38 รายการ demo, service type, sale team, overhead, FX rate, รายการมาตรฐาน Install/PMCM, MRC catalog) ไปเก็บใน ฐานข้อมูลกลาง Supabase โดยข้อมูลจริงจากไฟล์ Excel มีถึง 6,221 รายการ (active 5,382 · blocked 839) — แอปโหลดผ่าน REST อัตโนมัติ และยังเปิดแบบ offline ได้

สถาปัตยกรรมและลำดับแหล่งข้อมูล

index.html (เปิดจากไฟล์ / โฮสต์ที่ไหนก็ได้) │ ├─ 1) Supabase PostgREST ── GET /rest/v1/<table> (anon key, อ่านอย่างเดียว, แบ่งหน้า 1,000 แถว) │ └─ สำเร็จ → ใช้ทันที + เก็บลง cache ├─ 2) localStorage cache ── est_master_cache_v1 (เปิดเร็ว / ใช้ตอน offline) └─ 3) ชุด demo ฝังในไฟล์ ── ~38 รายการ (กรณียังไม่ตั้งค่า DB และไม่มี cache)

ขอบเขต Phase 1 (และที่ตัดออกโดยตั้งใจ)

ทำ: ตาราง master 11 ตาราง · seed จาก Excel · RLS อ่านสาธารณะ/เขียนเฉพาะ service role · loader ในแอป (item, service_type, sale_team, mrc_catalog, install_pmcm_standard, fx_rate)
ยังไม่ทำ (Phase 2): ตาราง transaction (ใบ EST, Quotation, approval log) · UI widget คงที่ (radio overhead, dropdown) render จาก DB · ระบบ login/สิทธิ์จริง · ไม่ลากตาราง ERP เต็ม 47 ตารางของ blueprint มาทั้งชุด — ใช้เฉพาะส่วนที่แอปใช้จริง

ข้อสมมติฐานที่บันทึกไว้

1. ราคาทุน = 68% ของ MSRP (Excel มีเฉพาะราคาขาย) — สูตรอยู่ฝั่งแอป ปรับได้ที่ applyMasterData()
2. รายการ blocked = true เก็บใน DB ครบ (ตาม lifecycle ของ blueprint) แต่แอปกรองออกจาก catalog
3. MRC catalog ไม่มีใน Excel rev นี้ — seed จากข้อมูลในแอป (ระบุ source ใน schema)
4. FX ใน Excel มี 9 ช่วง (ในแอปเดิม hardcode 8) — DB ใช้ตาม Excel เป็นหลัก

Transaction Layer — Customer & Quotation (migration #2: db/schema2_customer_quotation.sql)

ตาม blueprint M04 (Customer) + M08 (Sales) · RLS deny-by-default — publishable key อ่าน/เขียนไม่ได้ จนกว่าจะมี Auth (Phase 2.4) · ใช้รองรับการเชื่อม CRM

customer

M04 Customer Master — เจ้าของข้อมูลจริงคือ CRM (sync ผ่าน crm_ref)
  • customer_no 'CUST-YYMM-SEQ4' PK
  • name · tax_id (unique เมื่อมีค่า)
  • address · contact_name/email/phone
  • sale_team → sale_team.name
  • crm_ref (Account Id ฝั่ง CRM)
  • blocked · created_at · updated_at

quotation

M08 ใบเสนอราคา (header) — แทน object ใน localStorage
  • quote_no 'QT-YYYYMM-SEQ' PK
  • est_no (อ้างใบ EST ต้นทาง)
  • customer_no → customer
  • quote_type check(onetime/recurring/zoom)
  • status check(draft/sent/accepted/expired/cancelled)
  • quote_date · valid_until (check ≥ quote_date)
  • amount_ex_vat · discount · vat_pct · amount_inc_vat
  • pay_terms · salesperson · remark · crm_ref

quotation_line

บรรทัดรายการในใบเสนอราคา
  • quote_no + line_no PK ผสม
  • quote_no → quotation (on delete cascade)
  • item_no → item (nullable — บรรทัดอิสระ)
  • description · qty · unit_price · line_total
  • is_mrc (บรรทัดรายเดือน × term แล้ว)
  • trigger: ห้ามใช้ item ที่ blocked (BR-M08-006)

Transaction Layer — User & Approval (migration #3: db/schema3_user_approval.sql)

รองรับ approval process ตาม role model ในแอป (presale/sale/clevel/ceo) · RLS deny-by-default · ขอบเขต = เก็บ audit trail + สถานะ (การ route ตาม net margin คำนวณในแอป มีอยู่แล้ว) · seed ผู้อนุมัติจริง 14 คน

app_user

ผู้ใช้ระบบอนุมัติ — standalone ยังไม่ผูก Supabase Auth
  • user_id uuid PK (default gen_random_uuid)
  • auth_uid uuid unique (เผื่อ map auth.users ภายหลัง)
  • email (unique ตาม lower) · full_name
  • role check(presale/sale/clevel/ceo)
  • c_level_role check(COO/CPO/COS) — เฉพาะ clevel
  • sale_team → sale_team.name
  • title · active · created_at · updated_at
  • check: (role=clevel) ⇔ มี c_level_role

est_document

ใบ Estimated Cost Sheet — สิ่งที่ถูกอนุมัติ แล้วแปลงเป็น quotation
  • est_no 'EST-YYYYMM-SEQ4' PK
  • customer_no → customer
  • service_code → service_type (กำหนด threshold)
  • sale_team → sale_team · created_by → app_user
  • project_value · net_profit · net_margin_pct (snapshot)
  • required_level check(director/clevel/ceo)
  • status check(draft/pending/approved/rejected/converted/cancelled)
  • quote_no → quotation (set เมื่อแปลง)
  • payload jsonb (snapshot ฟอร์มทั้งใบ)
  • trigger: touch updated_at

approval_request

คำขออนุมัติ 1 รายการต่อการส่ง 1 ครั้ง
  • request_id bigserial PK
  • est_no → est_document (on delete cascade)
  • requested_by → app_user
  • net_margin_pct · required_level (snapshot)
  • current_step · total_steps
  • status check(pending/approved/rejected/cancelled)
  • opened_at · closed_at
  • unique partial: 1 คำขอ pending ต่อ est_no

approval_log

audit trail แต่ละก้าวในสายอนุมัติ
  • log_id bigserial PK
  • request_id → approval_request (on delete cascade)
  • step_no · step_role check(presale/sale/clevel/ceo)
  • actor_user → app_user · actor_name (snapshot)
  • action check(submitted/approved/rejected/returned/commented)
  • comment · acted_at

Business Rules

RuleคำอธิบายSeverity
BR-ITM-001item.blocked = true ห้ามปรากฏใน catalog ของแอป (กรองที่ loader) — ตรงกับ lifecycle Discontinued ของ blueprintError
BR-ITM-002item_no ห้ามซ้ำ (PK) — generator ตรวจ dup ก่อน insert (rev 2.4.7: dup = 0)Error
BR-SEC-001RLS เปิดทุกตาราง: anon key = SELECT เท่านั้น · INSERT/UPDATE/DELETE ทำได้เฉพาะ service role (dashboard / pipeline นำเข้า)Error
BR-FX-001fx_rate.end_date ≥ start_date (DB check constraint) · แอปเลือกแถว current จากวันที่ปัจจุบันError
BR-SVC-001appv_l1 ≥ appv_l2 (Director threshold ≥ C-Level) — ยกเว้น SI Connectivity (AM) ที่ l1 = 1.00 ตาม ReferenceWarning
BR-APP-001แอปต้องเปิดใช้งานได้แม้ DB ล่ม/offline — cache ล่าสุดใน localStorage, fallback สุดท้ายคือชุด demo ฝังในไฟล์Warning
BR-MIG-001การอัปเดต master ทำผ่าน generate_seed.py + seed.sql เท่านั้น (truncate + reload ทั้งชุด) — ห้ามแก้รายแถวปนกับ pipeline เพื่อให้ตรวจสอบกลับ Excel rev ได้Warning
BR-QT-001quotation_line.item_no ที่ blocked = true ถูกปฏิเสธที่ DB ด้วย trigger (ตาม BR-M08-006 ของ blueprint)Error
BR-QT-002ลูกค้า blocked = true สร้างใบเสนอราคาใหม่ไม่ได้ (trigger ที่ insert)Error
BR-QT-003valid_until ≥ quote_date · discount ≥ 0 · status/quote_type จำกัดด้วย check constraintError
BR-QT-004ตาราง customer/quotation/quotation_line = RLS deny-by-default — publishable key เข้าถึงไม่ได้ เข้าผ่าน service role (server side) เท่านั้น จนกว่าจะมี AuthError
BR-APV-001app_user.role จำกัด presale/sale/clevel/ceo ตรงกับ ADMIN_ROLES ในแอป · บังคับ (role=clevel) ⇔ c_level_role not nullError
BR-APV-0021 est_document มี approval_request ที่ status=pending ได้ไม่เกิน 1 (partial unique index) — กันส่งอนุมัติซ้ำซ้อนError
BR-APV-003est_document.net_margin_pct/required_level เป็น snapshot ที่แอปคำนวณตอนส่ง — DB เก็บเพื่อ audit ไม่ re-route เอง (audit-trail-only)Warning
BR-APV-004ตาราง app_user/est_document/approval_request/approval_log = RLS deny-by-default เช่นเดียวกับ migration #2Error
migration #3 เพิ่ม approval workflow แบบ audit-trail-only — กฎ immutable ledger (BR-M05) เต็มรูปแบบ และ RLS ราย role จะบังคับใช้เมื่อเปิด Supabase Auth (Phase 2.4) ผ่าน app_user.auth_uid = auth.uid()
